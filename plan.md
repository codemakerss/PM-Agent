# BA-Agent 项目启动计划

## 1. 项目目标

BA-Agent 的目标是构建一个面向 Business Analyst（BA）的智能代理系统，帮助 BA 阅读产品设计、功能文档、业务 Wiki、用户反馈、会议纪要、竞品资料和数据报告，并自动完成或辅助完成需求分析、利益相关方分析、PRD 生成、需求澄清、方案评估和交付跟踪。

长期目标不是只生成 PRD，而是逐步覆盖 BA 的核心工作流，使 Agent 能够在明确边界和人工审核机制下，承担大量 BA 日常分析与协调工作。

## 2. 核心业务理解

BA 在产品建设中的核心职责包括：

- 识别客户真实需求，而不是只记录客户提出的表面功能。
- 识别相关利益相关方，包括客户、业务方、产品经理、研发、设计、运营、客服、法务、财务、管理层等。
- 分析不同角色的目标、痛点、约束、风险和成功标准。
- 将模糊需求转化为可讨论、可验证、可开发、可验收的产品需求。
- 在客户价值、业务价值、技术可行性、实施成本和风险之间做平衡。
- 产出高质量 PRD、用户故事、验收标准、流程图、需求优先级、风险清单和沟通材料。

BA-Agent 应围绕这些职责设计，而不是仅作为文档生成工具。

## 3. 输入与输出定义

### 3.1 输入材料

项目初期需要准备以下输入类型：

- 产品设计文档：功能说明、原型说明、页面流程、交互说明。
- 功能 Wiki：已有系统能力、业务规则、接口说明、历史决策。
- 客户材料：客户访谈、用户反馈、工单、问卷、销售记录、客户成功记录。
- 业务材料：业务流程、组织结构、KPI、增长目标、合规要求。
- 研发材料：系统架构、技术限制、接口文档、数据模型、开发成本估算。
- 市场与竞品材料：竞品功能、行业报告、公开案例。
- 项目材料：会议纪要、需求池、路线图、Jira/Linear/Tapd 任务、版本计划。

### 3.2 标准输出

BA-Agent 应至少支持以下输出：

- 客户需求分析报告。
- 利益相关方地图。
- 用户画像与使用场景。
- 业务流程分析。
- 问题定义与机会点分析。
- 需求澄清问题列表。
- 需求优先级建议。
- PRD 初稿与 PRD 改进建议。
- 用户故事与验收标准。
- 功能范围说明。
- 非功能需求清单。
- 风险、依赖与假设清单。
- 方案可行性分析。
- 会议准备材料与会议纪要总结。
- 决策记录 ADR 或需求决策记录。
- 面向研发、设计、业务方、客户的不同版本沟通材料。

## 4. 项目准备工作

### 4.1 业务准备

- 明确目标用户：企业 BA、产品经理、解决方案顾问、客户成功、交付顾问。
- 明确首批场景：PRD 生成、需求澄清、客户需求分析、利益相关方分析。
- 收集真实样例文档：至少准备 20 到 50 组真实或脱敏的需求案例。
- 定义优秀 BA 输出标准：什么是好的需求分析、好的 PRD、好的用户故事。
- 建立 BA 工作流清单：从需求输入到上线验收，每一步需要哪些判断和产出。

### 4.2 数据准备

- 文档格式：Markdown、PDF、Word、Excel、HTML、Confluence、Notion、飞书文档等。
- 数据清洗：去除无关内容、处理重复文档、统一标题层级、保留来源信息。
- 文档切分：按章节、业务对象、功能模块、需求项进行结构化切分。
- 元数据设计：来源、作者、时间、业务域、产品线、版本、可信度、权限级别。
- 知识库建设：建立向量检索、关键词检索和结构化索引。
- 权限控制：不同角色只能访问对应客户、项目或产品线的资料。

### 4.3 技术准备

- 选择 Agent 框架：LangGraph、OpenAI Agents SDK、LlamaIndex Workflow、AutoGen 或自研轻量编排。
- 选择模型供应商：OpenAI、Anthropic、Google、国产模型或混合路由。
- 准备 RAG 能力：文档解析、Embedding、向量数据库、混合检索、重排序。
- 准备工具调用能力：搜索知识库、读取文档、生成图表、创建任务、写入文档系统。
- 准备评测框架：自动评测、人工评审、回归测试、案例集管理。
- 准备日志与可观测性：Prompt、工具调用、检索结果、模型输出、用户反馈。

### 4.4 产品准备

- 定义 MVP 范围。
- 定义用户操作入口：Web App、Chat UI、文档插件、Slack/飞书/企业微信 Bot。
- 定义交互方式：上传文档、选择业务场景、提出分析问题、生成报告、人工确认。
- 定义人工审核机制：Agent 产出必须标注假设、证据来源和不确定性。
- 定义结果导出格式：Markdown、Word、PDF、Confluence、Notion、Jira 用户故事。

## 5. Agent 能力设计

### 5.1 核心 Agent 类型

BA-Agent 可以拆分为多个专业 Agent：

- 文档理解 Agent：读取产品设计、Wiki、会议纪要，提取结构化信息。
- 客户需求 Agent：识别客户目标、痛点、使用场景、隐藏需求。
- 利益相关方 Agent：识别相关角色、诉求、影响力、冲突点和沟通策略。
- 需求澄清 Agent：发现信息缺口、矛盾、模糊表达，生成澄清问题。
- PRD Agent：生成 PRD、用户故事、验收标准和功能范围说明。
- 可行性 Agent：分析业务价值、技术复杂度、实施成本和风险。
- 优先级 Agent：基于 RICE、MoSCoW、Kano 等方法给出优先级建议。
- 评审 Agent：检查 PRD 是否完整、一致、可开发、可验收。
- 项目协作 Agent：生成会议材料、同步任务、追踪需求状态。

### 5.2 多 Agent 协作流程

典型流程：

1. 用户上传或指定产品文档、客户材料、业务 Wiki。
2. 文档理解 Agent 提取功能、业务流程、角色、限制条件。
3. 客户需求 Agent 分析客户目标、痛点和真实需求。
4. 利益相关方 Agent 识别各角色诉求与冲突。
5. 需求澄清 Agent 生成待确认问题。
6. 可行性 Agent 分析价值、成本、风险和依赖。
7. PRD Agent 生成结构化 PRD。
8. 评审 Agent 检查遗漏、矛盾和不可验收项。
9. 用户确认后导出文档或同步到项目管理系统。

## 6. Harness Engineering 计划

Harness Engineering 指围绕 Agent 建立可运行、可测试、可观测、可迭代的工程环境，避免 Agent 只是一次性 Prompt Demo。

### 6.1 Agent Harness

- 建立统一任务入口：输入任务类型、文档集合、用户目标和输出格式。
- 建立状态管理：记录每一步 Agent 的中间结果、证据、假设和决策。
- 建立工具注册机制：不同 Agent 可以调用受控工具。
- 建立错误恢复：检索失败、文档解析失败、模型输出不合格时自动重试或请求人工确认。
- 建立人工介入点：对高风险判断、需求变更和最终 PRD 设置审核节点。

### 6.2 Evaluation Harness

- 建立黄金案例集：真实或脱敏 BA 案例，包含输入资料和优秀人工输出。
- 定义评测维度：完整性、准确性、可追溯性、可执行性、业务洞察、风险识别、格式合规。
- 自动评测：使用 LLM-as-judge、规则校验、结构化字段校验。
- 人工评测：由 BA、PM、研发、业务方分别打分。
- 回归测试：每次 Prompt、模型、检索策略或工具变更后运行固定案例。
- A/B 测试：比较不同 Agent 流程、模型和 Prompt 的效果。

### 6.3 Data Harness

- 文档导入管道。
- 文档解析与清洗管道。
- Chunk 切分与 Embedding 管道。
- 向量库和关键词索引更新。
- 数据权限与脱敏处理。
- 来源引用与版本管理。

### 6.4 Prompt Harness

- Prompt 模板管理。
- 角色指令、任务指令、输出格式分离。
- Prompt 版本控制。
- Prompt 单元测试。
- Prompt 与评测结果关联。
- 支持中文、英文或双语输出。

## 7. 推荐技术架构

### 7.1 MVP 架构

- 前端：Next.js 或 Streamlit。
- 后端：Python FastAPI。
- Agent 编排：LangGraph 或 OpenAI Agents SDK。
- 文档解析：Unstructured、MarkItDown、PyMuPDF、python-docx、openpyxl。
- 向量数据库：Chroma、Qdrant 或 pgvector。
- 数据库：PostgreSQL。
- 文件存储：本地文件系统、S3 或 MinIO。
- 任务队列：Celery、RQ 或 Dramatiq。
- 可观测性：LangSmith、OpenTelemetry、自建日志表。

### 7.2 生产架构

- 用户权限系统。
- 多租户知识库。
- 文档版本管理。
- Agent 工作流编排服务。
- 评测与反馈服务。
- 审计日志。
- 成本监控。
- 模型路由与降级策略。
- 企业文档系统集成。
- Jira、Confluence、Notion、飞书、企业微信、Slack 集成。

## 8. 功能路线图

### Phase 0：项目定义与样例准备

目标：明确 BA-Agent 的边界、用户和评测标准。

需要完成：

- 定义目标用户和首批业务场景。
- 收集并脱敏样例文档。
- 定义 PRD、需求分析和利益相关方分析模板。
- 建立 10 个最小评测案例。
- 确定技术栈和项目目录结构。

交付结果：

- 项目范围说明。
- 样例数据集。
- 输出模板。
- 初版评测标准。
- 技术选型文档。

### Phase 1：MVP 文档分析与 PRD 生成

目标：让用户上传文档后，Agent 能生成可用的 PRD 初稿。

需要完成：

- 文档上传与解析。
- RAG 知识库。
- 基础 Chat UI。
- 文档理解 Agent。
- PRD Agent。
- 来源引用。
- Markdown 导出。

交付结果：

- 可运行 MVP。
- 支持基于文档生成 PRD。
- 支持引用原始材料。
- 支持人工修改和重新生成。

### Phase 2：客户需求与利益相关方分析

目标：让 Agent 不只是整理文档，而是能进行 BA 级分析。

需要完成：

- 客户需求 Agent。
- 利益相关方 Agent。
- 需求澄清 Agent。
- 业务流程提取。
- 痛点、目标、约束、风险分析。
- 利益冲突与沟通策略分析。

交付结果：

- 客户需求分析报告。
- 利益相关方地图。
- 需求澄清问题列表。
- 业务流程摘要。

### Phase 3：评审、优先级与可行性分析

目标：提升 Agent 输出质量，使其能辅助需求决策。

需要完成：

- PRD 评审 Agent。
- 可行性分析 Agent。
- 优先级 Agent。
- 需求冲突检测。
- 缺失需求检测。
- 非功能需求检查。
- 自动评测 Harness。

交付结果：

- PRD 质量评分。
- 风险和依赖清单。
- 优先级建议。
- 回归评测报告。

### Phase 4：项目协作与工具集成

目标：让 BA-Agent 融入真实工作流。

需要完成：

- Jira/Linear/Tapd 任务生成。
- Confluence/Notion/飞书文档同步。
- 会议纪要总结。
- 决策记录生成。
- 需求状态跟踪。
- 多角色评论与审批。

交付结果：

- 从需求分析到任务创建的完整流程。
- 支持团队协作和审批。
- 支持版本管理和审计。

### Phase 5：接近替代 BA 工作流

目标：覆盖 BA 的主要日常工作，让 Agent 成为自动化 BA 助手。

需要完成：

- 主动发现需求风险。
- 主动提出客户访谈问题。
- 自动准备需求评审会材料。
- 自动追踪需求变更影响。
- 自动比较不同客户需求的共性与差异。
- 自动生成产品路线图建议。
- 自动识别交付后反馈和下一轮优化机会。

交付结果：

- BA 工作台。
- 多项目、多客户、多产品线分析。
- Agent 主动任务推荐。
- 可审计的需求决策链路。

## 9. PRD 输出模板建议

标准 PRD 应包含：

- 背景与目标。
- 问题定义。
- 客户与用户画像。
- 利益相关方分析。
- 使用场景。
- 当前流程与目标流程。
- 功能范围。
- 用户故事。
- 详细需求。
- 非功能需求。
- 数据需求。
- 权限与角色。
- 边界情况。
- 验收标准。
- 风险、依赖与假设。
- 埋点与成功指标。
- 发布计划。
- 待澄清问题。

## 10. 评测标准

### 10.1 输出质量指标

- 完整性：是否覆盖关键业务背景、用户、流程、功能和验收条件。
- 准确性：是否与输入材料一致，是否避免无依据编造。
- 可追溯性：重要结论是否能引用来源。
- 可执行性：研发、设计、测试是否能根据输出开展工作。
- 业务洞察：是否能发现隐藏需求、冲突、机会点和风险。
- 结构化程度：是否便于阅读、评审和导入工具。
- 可验证性：验收标准是否明确。

### 10.2 Agent 行为指标

- 检索命中率。
- 引用准确率。
- 澄清问题质量。
- 幻觉率。
- 重试成功率。
- 人工修改比例。
- 用户采纳率。
- 单次任务成本。
- 输出耗时。

## 11. 风险与应对

- 风险：Agent 编造不存在的业务规则。
  应对：强制来源引用，区分事实、推断和假设。

- 风险：输入文档质量差导致分析错误。
  应对：先做文档质量评分和缺口识别。

- 风险：PRD 看起来完整但不可开发。
  应对：加入验收标准、边界条件、非功能需求和研发评审 Agent。

- 风险：不同利益相关方诉求冲突。
  应对：输出冲突矩阵和决策建议，不直接隐藏冲突。

- 风险：企业数据安全和权限问题。
  应对：多租户隔离、权限控制、脱敏、审计日志。

- 风险：Agent 无法真正替代 BA 的判断。
  应对：先定位为 BA Copilot，再逐步自动化低风险、高重复任务。

## 14. 成功标准

MVP 成功标准：

- 用户可以上传或选择业务文档。
- Agent 可以识别客户需求、角色、功能点和待澄清问题。
- Agent 可以生成结构清晰、可编辑的 PRD 初稿。
- PRD 中的重要结论可以追溯到输入材料。
- 人工 BA 认为输出可以节省至少 30% 的初稿准备时间。

中长期成功标准：

- Agent 可以在多个项目中复用业务知识。
- Agent 可以主动发现需求缺口和利益冲突。
- Agent 可以生成研发、设计、测试和业务方都能使用的材料。
- Agent 可以与项目管理和文档系统集成。
- Agent 可以承担 BA 日常工作中 50% 以上的重复性分析和文档工作。

## 15. 下一步建议

建议下一步先完成 Phase 0：

1. 定义 MVP 的首个具体业务场景。
2. 收集 10 组脱敏样例文档。
3. 确定 PRD 与需求分析输出模板。
4. 选择技术栈。
5. 创建项目基础代码结构。

完成这些准备后，再进入 MVP 开发。这样可以避免直接写 Agent Demo，但没有数据、没有评测、没有真实业务标准，最终难以成为可用产品。
